# 改修仕様インテーク（Loop Engineering 運用ガイド）

本ディレクトリは、機能追加・改修を Loop Engineering で回すときの**仕様の受け付け口**である。

> 前提とする開発観: 仕様を与えると、まずその仕様に対応する**決定的な品質ゲート**（テスト・契約検査・lint）が追加され、
> ゲートが緑になるまで実装ループが回る。仕様は「実装の指示書」ではなく「ゲートの原料」である。

- 仕様の書き方（テンプレート）: [`templates/`](./templates/)
- **記入済みサンプル（架空の依頼・全体像はまずこれを読む）**: [`examples/CR-20260706-discount-approval/`](./examples/CR-20260706-discount-approval/)
- 全依頼共通の既定（毎回書かない決まりごと）: [`defaults.md`](./defaults.md)
- 記入済み仕様の置き場所: `requests/<依頼ID>/`
- ドメインの型・behavior の SSoT: [`../dsl/domain-model.md`](../dsl/domain-model.md)（Stage 1。本ディレクトリの仕様が合意されたらエンジニア/AI が反映する）
- API 契約の SSoT: `apps/api-fsharp/openapi.yaml`

参考元: [domain_spec_template_pack](https://github.com/geibee/domain_spec_template_pack)（version1 のカード形式と version2 の制約 4 分類・責務マトリクスを本リポジトリ向けに簡略化した）。

## 1. 改修の標準手順

```
①起票・合意 → ②仕様PR → ③ゲート先行(Red) → ④実装ループ(Green) → ⑤マージ・監視
```

### ① 起票・合意形成（人間: 顧客 + 起票者）

1. `requests/<依頼ID>/`（例: `requests/CR-20260710-discount-approval/`）を作り、
   [`templates/00_change-request.md`](./templates/00_change-request.md) をコピーして記入する。
2. 変更種別に応じて追加テンプレート（§2 の対応表）をコピーして記入する。
3. **例・反例・境界例は必ず顧客とレビューし、承認欄を埋める。** ここが受入基準そのものであり、
   後工程のテストオラクルになる。用語・状態遷移・判定表も顧客の言葉のまま書く（実装用語に翻訳しない）。
4. 毎回同じになる決まりごと（エラー表示・競合時の挙動・書式・a11y 等）は記入しない。
   [`defaults.md`](./defaults.md) の既定に従い、**既定から外れる場合だけ**理由付きで書く。

### ② 仕様 PR（人間 or AI 起案、人間承認）

- `specs/requests/<依頼ID>/` のみを含む PR を作る。**実装・DSL・openapi.yaml の変更は含めない**
  （DSL↔コード整合ゲートや生成コードドリフト検査を赤にしないため。それらは③以降の実装ブランチで先頭コミットにする）。
- レビュー観点: 必須セクションが埋まっているか / 例・反例に顧客承認があるか / 未決事項が「仮決定+決定者+期限」の形か。

### ③ ゲート先行（AI セッション、Red フェーズ）

改修セッション（Claude Code / ralph-orchestrator タスク）に依頼ディレクトリを渡し、**先にゲートだけ**追加させる:

1. `dsl/domain-model.md` に型・behavior 差分を反映（→ `dotnet build` の網羅性検査が新ケースで赤になる）
2. `apps/api-fsharp/openapi.yaml` に契約差分を反映 + クライアント再生成（→ Spectral / oasdiff / ドリフト検査 / zod スキーマが効く）
3. 各テンプレート末尾の「品質ゲート化対応表」に従い、**失敗するテスト**を追加（テスト ID を発行し、仕様の該当行に転記する）
4. この時点で `scripts/verify.sh` は赤であること（赤にならない = ゲートが仕様を検査できていない）

> **契約差分の承認（人間チェックポイント）**: 手順 2 の `openapi.yaml` 差分（および非同期連携のイベントスキーマ差分）は
> AI の起案のままにせず、**エンジニアが承認してから④へ進む**。
> 新規エンドポイントまたは破壊的変更を含む場合は、契約差分コミットを push した時点でレビューを依頼し、承認まで実装を凍結する。
> 既存契約と互換な追加のみの場合は、PR レビューで契約差分コミットを重点確認する運用でよい。
> 理由: 契約はフロント生成コード・Pact・外部利用者に波及し、後からの改名は破壊的変更になる。
> Spectral / oasdiff が機械的に守るのはスタイルと互換性だけで、リソース設計・命名の適切さ（DEF-API-04）は人間の責務。
> 承認結果は各 API 仕様の「契約差分の承認」欄に記録する。

### ④ 実装ループ（AI セッション、Green フェーズ）

- `scripts/verify.sh` が緑になるまで実装する。ralph-orchestrator を使う場合は `.ralph/tasks.toml` にタスク分割する。
- 仕様に書かれていない設計判断（コンポーネント分割・hook・DB スキーマ・F# モジュール構成）は
  `AGENTS.md` の規約とゲートの範囲内で AI が自律決定してよい。
- **仕様と矛盾・欠落を見つけたら実装で埋めず、依頼の「未決事項」に差し戻して質問する。**

### ⑤ マージ・監視

- verify（軽量ゲート）緑 + レビューでマージ。nightly（重量ゲート）の結果と `LESSONS.md` を確認し、
  仕様起因の教訓は該当依頼の仕様に反映する（仕様変更は [`templates/00_change-request.md`](./templates/00_change-request.md) の変更履歴に追記）。

## 2. 変更種別とテンプレート対応表

`00_change-request.md` は**全依頼で必須**。以下は該当するものだけ追加する（1 依頼に複数可）。

| 変更種別 | テンプレート | 典型例 |
|---|---|---|
| ドメインルール・状態遷移・業務判定 | [`10_domain-behavior.md`](./templates/10_domain-behavior.md) | 値引き承認フロー追加、ロットに新状態追加 |
| バックエンド オンライン処理（API） | [`20_api-endpoint.md`](./templates/20_api-endpoint.md) | エンドポイント追加・入出力項目変更 |
| バッチ処理 | [`30_batch.md`](./templates/30_batch.md) | 月次締め、日次の在庫棚卸 |
| 非同期処理・イベント連携 | [`40_async-integration.md`](./templates/40_async-integration.md) | 出荷完了イベントの外部通知、Outbox 経由連携 |
| フロントエンド ページ | [`50_frontend-page.md`](./templates/50_frontend-page.md) | 画面追加、一覧に列・フィルタ追加 |
| フロントエンド 共通コンポーネント | [`60_frontend-component.md`](./templates/60_frontend-component.md) | 状態バッジ、承認ダイアログの共通化 |
| 純粋関数・計算ロジック | [`70_pure-function.md`](./templates/70_pure-function.md) | 査定額の按分計算、調整率の適用規則 |
| 非機能要件 | [`80_nfr.md`](./templates/80_nfr.md) | 一覧 3 秒以内、同時 10 ユーザー |

**典型的な機能追加は `00 + 10 + 20 + 50` の 4 点セット**になる
（業務ルール → API → 画面。計算が絡めば 70 を足す）。

> 迷ったときの指針（制約の置き場所と表現形式 — 同じルールを二つの形式で書かない）:
> - データ・状態の**形そのもの**（何と何を持つか、どんな種類があるか）→ `10` の「型スケッチ」（関数型ドメインモデリング記法）
> - 単一の値だけで判定できる制約（0 以上、桁数、形式）→ `10` の「値の制約」
> - 状態によって操作の可否が変わる → `10` の「状態遷移表」
> - 複数条件の組み合わせで結果が変わる → `10` の「判定表」
> - 誰ができる操作か → `10` の「権限マトリクス」（他のテンプレートには書かない）
> - DB・他の案件・権限・現在日時に依存する制約 → `10` の「外部参照・コンテキスト依存ルール」
> - 数式・計算 → `70`
> - 画面上の表示・入力の都合 → `50` / `60`

## 3. 責務の線引き — 顧客と合意する範囲 / 委ねる範囲

「システムのプロでない顧客と合意形成できる範囲」を仕様の上限とする。
それより下のレイヤは AI・エンジニアがゲートと規約の範囲で決める。

### 顧客と合意して仕様に**必ず**書くこと（漏れ防止チェックリスト）

- [ ] **用語**: 新しい業務用語の意味、既存用語との違い、使ってはいけない紛らわしい語
- [ ] **状態と遷移**: どの状態から何ができて何ができないか（**禁止も明示**）、後戻り・取消の可否
- [ ] **判定ルール**: 条件の組み合わせと結果、複数該当時の優先順位、どれにも該当しないときのデフォルト
- [ ] **値の制約**: 範囲・単位・桁・端数の扱い（切り捨て/四捨五入/エラー）
- [ ] **エラー時の業務的扱い**: 利用者に何と伝えるか、利用者は次に何をすればよいか
- [ ] **例・反例・境界例**: 代表的な正常例 / やってはいけない例 / 境界ちょうど・±1 の例（**顧客承認必須**）
- [ ] **権限**: 誰が実行できる操作か、できない人が試みたときの扱い
- [ ] **画面**: 表示・入力する業務項目と必須性、状態ごとに出る/出ないボタン、操作後どこへ行くか
- [ ] **バッチ**: 業務上いつ動くべきか、二重実行・再実行してよいか、失敗したら業務はどうするか
- [ ] **非同期**: どれくらいの遅れまで許容か、順序の入れ替わり・重複が起きたら業務的にどうなるか
- [ ] **規模と性能の期待**: 件数の桁（百・万・百万）、体感の許容（「画面は数秒」「締めは朝までに」）
- [ ] **既存データ・互換性**: 改修時点で存在するデータをどう扱うか（最重要の頻出漏れ）

### AI が起案し、**エンジニアが承認する**もの（中間層）

顧客レビューは不要だが、決定的ゲートでは品質を担保しきれないため AI 任せにもしない。
①〜②の段階でエンジニアが直接書いてもよいし、③で AI に起案させて承認してもよい（承認の記録は必須）。

- API インターフェース（URL・リソース設計・JSON 形状・HTTP ステータス割り当て）= `openapi.yaml` 差分 → ③の契約差分承認
- 非同期連携のイベントスキーマ（物理形式）→ 同上
- `dsl/domain-model.md` への型・behavior 翻訳（`10` の型スケッチが下書きになる）
- エラーコードの採番・ProblemDetails への割り当て

### 仕様に**書かない**こと（AI がゲートと規約の範囲で決める）

- コンポーネント分割・カスタム hook・props 設計・使用ライブラリ
- DB スキーマ・SQL・インデックス
- F# のモジュール構成・エラー型の内部構造（命名は `AGENTS.md` の英訳規約に従い機械的に決まる）
- テスト技法の選択（例示 / PBT / E2E の割り当てはゲート化対応表が既定する）

> 例外: 上記を**あえて指定したい**場合（既存画面との統一など）は `00` の「制約・指定事項」に理由付きで書く。
> 理由のない実装指定はレビューで削除してよい。

## 4. 仕様 → 品質ゲート対応表（全体マップ）

各テンプレート末尾に詳細版がある。ここでは全体の対応だけ示す。

| 仕様セクション | 追加される決定的ゲート | 実行機構 |
|---|---|---|
| DSL 差分（型・behavior） | コンパイル + FS0025 網羅性（match 漏れが赤になる）、DSL↔コード整合検査 | `dotnet build --warnaserror` / verify |
| 状態遷移表 | 全遷移(禁止含む)のテーブルテスト + ステートフル PBT | dotnet test (FsCheck) |
| 判定表 | 全ルールの例示テスト + PBT（重なり・未一致検出） | dotnet test / Vitest |
| 値の制約 | SmartConstructor 境界値テーブルテスト + PBT (soundness) | dotnet test |
| 例・反例・境界例 | API 統合テスト / MSW コンポーネントテスト / E2E 代表導線 | dotnet test / Vitest / Playwright |
| API 契約差分 | Spectral lint、oasdiff 破壊的変更検査、Schemathesis、Pact、生成コードドリフト、認可マトリクス | verify / ci.sh |
| 画面: 項目・状態×操作マトリクス | `FE-MATRIX-*` 等の ID 付きテスト、カバレッジラチェット | Vitest + Testing Library |
| 画面: a11y | jest-axe / @axe-core/playwright | verify / nightly |
| バッチ: 冪等性・再実行 | 2 回実行 = 1 回のテスト、境界日付テスト、DB 統合テスト | dotnet test (Testcontainers) |
| 非同期: 配信保証 | Outbox at-least-once property、冪等キー検査、LocalStack 統合 | dotnet test / nightly |
| 非機能 | k6 / size-limit / Lighthouse / ZAP ルール / EXPLAIN 検査 | nightly |

> **注**: 認可マトリクス・スキーマ全数検証・バックエンドカバレッジラチェット・fast-check・a11y・k6 等は
> Issue [#9](https://github.com/geibee/sales-management/issues/9) の品質ゲート拡充で導入されるものを含む。
> 未導入のゲートに依存する仕様欄は、③の時点で「ゲート未導入」とテスト ID 欄に明記し、通常テストで代替する。

## 5. ID・トレーサビリティ規約

- **依頼 ID**: `CR-<YYYYMMDD>-<slug>`（例: `CR-20260710-discount-approval`）
- **仕様項目 ID**: テンプレート内の表の行に振る。`<依頼ID略記>-<種別>-<連番>`
  （例: 状態遷移 `DA-ST-01`、判定 `DA-DT-01`、例 `DA-EX-01`）
- **テスト ID**: 既存体系に従う（フロントは `docs/frontend-component-page-test-plan.md` の `FE-*` 体系に追記、
  バックエンドはテスト名に仕様項目 ID を含める）。③でゲートを追加したら**仕様の該当行にテスト ID を転記**し、
  仕様 ↔ ゲートの対応を双方向にする。

## 6. 仕様のライフサイクルと SSoT（腐らせないための規約）

`requests/` は**依頼時点の合意スナップショット**であり、恒久的な SSoT ではない。

- 依頼の状態は `draft → review → approved → implementing → done` で遷移し、**done になった依頼は凍結する**（以後編集しない。監査・経緯確認用）
- 実装完了後の恒久的な真実は、`dsl/domain-model.md`（型・behavior）/ `openapi.yaml`（契約）/ テストコード（例・境界）/ `defaults.md`（共通既定）に宿る
- 完了済み領域を再改修するときは、古い依頼の記述ではなく**現行のコード・契約・テストを正**として新しい依頼を起票する

これを守らないと `requests/` が「二つ目の、腐っていく SSoT」になる。

## 7. ディレクトリ構成

```
specs/
├── README.md                 # 本ガイド
├── defaults.md               # 共通既定仕様（一度だけ合意し、各仕様は逸脱のみ書く）
├── templates/                # 記入用テンプレート（コピーして使う。直接編集しない）
├── examples/                 # 記入済みサンプル（架空の依頼。書き方の参照用）
└── requests/                 # 依頼ごとの仕様一式
    └── CR-<YYYYMMDD>-<slug>/
        ├── 00_change-request.md      # 必須
        ├── 10_domain-behavior.md     # 該当時
        ├── 20_api-endpoint.md        # 該当時
        └── ...
```
