# アジャイル運用プレイブック — セレモニー別にやることリスト

> **いつ使うか**: PO（またはそれに相当する顧客側の意思決定者）が随時判断でき、スコープを調整できる開発。
>
> **方針**: 仕様を前倒しで書き溜めない。仕様 = **ストーリー + 顧客承認済みの例**（Specification by Example）まで縮退させ、
> 網羅は AI がやる — AI がテンプレートへ機械展開し、人間は **AI が挙げた問い（未決事項）に答えるだけ**。
> 人間待ちの凍結ポイントは契約承認（`20` §6 / `40` §7）ただ一つに絞り、トラック B / C のストーリーは人間待ちなしで完走させる。

## バックログリファインメント（スプリント開始前）

| # | やること | 担当 |
|---|---|---|
| R-1 | 各ストーリーにトラック判定（[`README.md`](./README.md) §0）を付ける。迷ったら A に倒す | PO + エンジニア |
| R-2 | トラック A / B: PO はストーリー 1 件につき**目的 1〜3 行 + 受入例 3〜7 個**だけ書く（`00` §1・§4 相当。所要 30 分以内。これ以上書かない） | PO |
| R-3 | AI がそれを該当テンプレートへ機械展開する（境界値・禁止遷移・権限マトリクス・エラー表の候補を埋める）。判断できない欄は「選択肢 + 仮案」の形で Q 表に**逆質問**として積む | AI |
| R-4 | PO は Q に答えるだけ。答えられない Q は「仮決定 + 期限」でスプリントに持ち込む（Ready を止めない） | PO |
| R-5 | トラック A は展開結果の表（遷移・権限・例）に PO の承認 ✔ を付ける | PO |

**Ready の定義**: トラック判定済み / 受入例あり / A は `10` の表と例に承認 ✔ / 未決 Q は全件期限付き。

## 仕様負債の扱い

- `requests/` はスナップショットであり、スプリントを跨いで育てない（done で凍結 — README §6）。恒久の真実は dsl / openapi / テスト / defaults に置く。
- AI 自律領域（内部設計）のドリフトはカバレッジラチェットと characterization テストが押さえる。仕様文書側のメンテナンス作業は発生させない。
- 完了済み領域を再改修するストーリーは、古い CR ではなく**現行のコード・契約・テストを正**として起票する。
