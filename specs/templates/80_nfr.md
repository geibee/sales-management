# 非機能要件仕様

> 適用対象: 性能・セキュリティ・可用性・運用性・データ保全の要求。
> **必ず「業務の事実 + 数値」で書く**（「速く」「安全に」は仕様にならない）。測定できない要求はゲート化できず、ループで守れない。
>
> **記入と承認の責任**: 全欄 AI が起案してよい（現状の計測値から要求値の叩き台を出す）。承認なしに approved にできない欄 —
> §1 業務上の根拠（規模の前提）・§2 要求値・§3 測定方法 = **顧客**。

## メタ情報 ★

| 項目 | 記入 |
|---|---|
| 依頼 ID / 仕様 ID | CR-… / NFR-01 |
| 分類 | 性能 / セキュリティ / 可用性 / 運用性 / データ保全 |
| 対象範囲 | 例: ロット一覧画面 / 全 API / 月次締めバッチ |

## 1. 業務上の根拠 ★

なぜこの要求か。規模の前提（件数の桁・利用者数・利用時間帯）を書く。

> 記入例: 月末 2 営業日は営業担当 10 名が同時にロット一覧と案件詳細を頻繁に開く。ロットは年間約 3 万件増える。5 年で 15 万件を想定。

## 2. 要求値 ★（分類ごとに該当表を使う）

### 性能

| 対象 | 条件（データ量・同時数） | 要求値 | 超過時の扱い |
|---|---|---|---|
| ロット一覧 API | 15 万件、同時 10 リクエスト | p95 で 3 秒以内 | 警告 → 改善タスク / 即エラー |
| 画面初期表示 | 通常回線 | 初期 JS を現状から増やさない（サイズ予算） | |

### セキュリティ

| 対象 | 要求 |
|---|---|
| 例: 値引き承認操作 | 操作者・日時・変更前後値の監査記録を残す |
| 例: 申請理由欄 | 出力時エスケープ（既定の ZAP / CSV インジェクション検査に含まれる） |

### 可用性・運用性

| 観点 | 要求 |
|---|---|
| 停止許容 | 例: 夜間バッチ時間帯以外は計画停止しない |
| 監視 | 例: バッチ失敗は当日中に気づける |

### データ保全

| 観点 | 要求 |
|---|---|
| 保持期間・削除 | |
| 移行時の検証 | 例: 移行前後で件数・金額合計が一致 |

## 3. 測定方法の合意 ★

| 項目 | 記入 |
|---|---|
| どの環境で測るか | CI（nightly）/ 本番相当環境 |
| データの作り方 | 例: シードスクリプトで 15 万件生成 |
| 継続監視か一度きりか | 継続（ゲート化）/ リリース前の一度きり |

## 品質ゲート化対応表（③で AI が記入）

> 非機能ゲートの多くは nightly 配置（実行時間が長いため）。Issue #9 Tier3 の導入状況に依存する項目は状態欄に明記する。

| 仕様欄 | 追加するゲート | 配置 / 状態 |
|---|---|---|
| API 性能 | k6 / NBomber の p95 閾値ゲート ※Issue #9 §11 | nightly |
| クエリ規模耐性 | 大量データ投入 + EXPLAIN の seq scan 検出 ※Issue #9 §11 | nightly |
| 画面サイズ予算 | size-limit（initial JS / chunk 上限）※Issue #9 §7 | verify |
| 画面性能 | Lighthouse CI（LCP/TBT）※Issue #9 §7 | nightly |
| セキュリティ | ZAP ルール / gitleaks / 認可マトリクス / 監査ログの統合テスト | verify + nightly |
| データ保全 | 移行スクリプトの前後検証テスト、スキーマスナップショット | verify |
