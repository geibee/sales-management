# 非同期処理・イベント連携仕様

> 適用対象: 「あとで反映されればよい」処理、他システム・他機能へのイベント通知（SQS / EventBridge / Outbox 経由）。
> 同期 API で足りるなら使わない。非同期にする理由（利用者を待たせない・外部が遅い/不安定）を §1 に書けないなら同期にする。

## 1. トリガと非同期にする理由 ★

| 項目 | 記入 |

| 非同期にする理由 | 例: 外部システムの応答が遅く、出荷操作を待たせたくない |

## 3. 遅延の許容 ★

> 記入例: 通常は数秒〜1 分。障害時でも 24 時間以内に届けば業務上問題ない（翌朝の売上集計までに揃えばよい）。

## 4. 順序と重複 ★（非同期仕様の最重要欄。「起きない」ではなく「起きたらどうするか」を書く）

| 観点 | 業務的な扱い |
|---|---|
| 順序の入れ替わり（後のイベントが先に届く） | 例: 同一ロットのイベントは順序が保たれる必要がある / 順不同でよい（理由: ） |
| 同じイベントが 2 回届いた | 例: 2 回目は無視される（出荷実績が二重計上されないこと）← 冪等キー: ロット番号+イベント種別 |
| 発生源では起きたのにイベントが消えた | 許容しない（at-least-once 必須。DB 更新とイベント記録は同時に確定する = Outbox） |

## 5. 失敗時の扱い ★

| 観点 | 記入 |
|---|---|
| 再試行 | 何度・いつまで粘るか（業務上の上限。例: 24 時間） |
| あきらめた後（DLQ 相当） | 誰が気づき、どう手当てするか（通知先の既定: DEF-OPS-01） |
| 受け手の処理が失敗し続ける間 | 他のイベント処理を止めてよいか（毒メッセージの隔離要否） |

## 6. 受け手が外部システムの場合（該当時）
