# API 仕様（バックエンド オンライン処理）

> 適用対象: エンドポイントの追加、入出力項目の追加・変更。
> URL・JSON 形状・HTTP ステータスへの翻訳（`openapi.yaml` 更新）はエンジニア/AI の仕事。ここには業務項目レベルで書く。
> 裏の業務ルールは `10_domain-behavior.md` に書き、ここからは参照だけする（二重管理しない）。

## メタ情報 ★

| 項目 | 記入 |
|---|---|
| 依頼 ID / 仕様 ID | CR-… / API-01 |
| 種別 | 参照 / 登録 / 更新 / 削除 / 操作（状態遷移の実行） |
| 参照するドメイン仕様 | DA-01 の DA-ST-01〜03 |

## 1. ユースケース ★

| 項目 | 記入 |
|---|---|
| 誰が（アクター） | 営業担当 |
| いつ・何のために | 査定済み案件の詳細を見ながら値引きを申請する |
| 事前条件 | 対象案件が「査定済み」であること |
| 成功時の事後条件 | 案件が「値引き承認待ち」になっている |
| 失敗時の事後条件 | 案件は一切変化していない |

## 2. 入力項目 ★（業務項目レベル）

| 項目 | 意味 | 必須 | 制約（DA 参照可） |
|---|---|---|---|
| 対象案件 | 値引きする販売案件 | ○ | 存在し「査定済み」であること |
| 値引き率 | | ○ | DA-01 §4 |
| 申請理由 | 上長が見るメモ | ○ | 500 文字以内 |

## 3. 出力項目 ★

| 項目 | 意味 |
|---|---|
| 更新後の案件状態 | 「値引き承認待ち」 |
| 承認後の想定契約額 | 参考表示用 |

## 4. 権限 ★

| 観点 | 記入 |
|---|---|
| 実行できる役割 | 営業担当、営業管理者 |
| 未ログイン時 | 実行不可（ログインを促す） |
| 権限がない役割が実行したとき | 実行不可（権限がない旨を表示） |

## 5. 同時実行・競合 ★（更新系のみ）

他の利用者が同じデータを先に変更していた場合の業務的扱い（本リポジトリは楽観ロック方式）。

> 記入例: 競合時はエラーとし、利用者は最新を取り直して再操作する（DA-ER-02）。自動マージはしない。

## 6. 互換性 ★

| 観点 | 記入 |
|---|---|
| 既存 API 利用箇所への影響 | なし（新規） / あり: |
| 破壊的変更（必須項目追加・項目削除・意味変更）か | いいえ / はい（理由と移行方針: ） |

## 7. 性能の期待（任意・数値があれば）

> 記入例: 対象案件は多くて数千件規模。通常操作として数秒以内に完了すればよい。
