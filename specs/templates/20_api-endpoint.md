# API 仕様（バックエンド オンライン処理）

> 本書の責務は「**業務項目レベルの入出力**と**契約差分の承認記録**」に限定する。他の関心事は書かない:
> - 業務ルール・状態遷移・権限 → `10_domain-behavior.md`（参照だけする。二重管理しない）
> - 競合制御・エラー形式・認証の既定 → `specs/defaults.md`（外れる場合のみ本書に理由付きで書く）
> - 性能要件 → `80_nfr.md`
>
> URL・JSON 形状・HTTP ステータスへの翻訳（`openapi.yaml` 差分)は AI が起案してよいが、
> **エンジニアの承認（§6）を得るまで実装に進まない**（`specs/README.md` ③参照）。
>
> **記入と承認の責任**: 全欄 AI が起案してよい。承認なしに先へ進めない欄 —
> §2 入力・§3 出力（業務項目の過不足）= **起票者/顧客**、§5 互換性の判断と §6 契約差分 = **エンジニア**。

## 4. 既定からの逸脱（該当時のみ・理由必須）

競合制御（DEF-API-02）・エラー形式（DEF-API-01)・認証認可の挙動（DEF-API-03）で既定と異なる点があれば書く。なければ「なし」。

## 5. 互換性 ★

| 観点 | 記入 |
|---|---|
| 既存 API 利用箇所への影響 | なし（新規） / あり: |
| 破壊的変更（必須項目追加・項目削除・意味変更）か | いいえ / はい（理由と移行方針: ） |

## 6. 契約差分の承認 ★（③ゲート先行フェーズで記入。エンジニアの人間レビュー必須）

`openapi.yaml` 差分のレビュー観点: リソース設計・命名が既存契約と一貫しているか（DEF-API-04）、
破壊的変更が §5 の申告と一致するか、エラー応答が ProblemDetails 規約に沿うか。

| 項目 | 記入 |
|---|---|
| openapi.yaml 差分（コミット / PR リンク） | |
| 差分の種類 | 新規エンドポイント / 互換な追加 / **破壊的変更** |
| エンジニア承認（承認者・日付） | 新規・破壊的変更は**承認まで実装凍結** |
