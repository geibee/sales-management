# ウォーターフォール運用プレイブック — フェーズ別にやることリスト

> **いつ使うか**: スコープ・納期・受入基準が契約で固定されており、仕様の前倒し合意が避けられない開発。
>
> **方針**: 前倒しで凍結するのは**外から観測できる面だけ**（`10` の表と例 / API・イベント契約 / `50` のマトリクス / `80` の数値）。
> 従来最も工数を食っていた**詳細設計フェーズ（どのコンポーネント・どのテーブル）は置かず**、AI がゲートの範囲で決める。
> 結合・総合テストも独立フェーズにしない — 設計時点で書いた E2E / NFR ゲートが CI で最初から回り続けるため、
> 「結合で発覚」が構造的に起きない。
>
> 人間は仕様を**書く**のではなく、AI 起案を**レビュー会で承認する**（記述と承認のコスト差を使う）。

## フェーズ 1: 要件定義

| # | やること | 担当 | 使う文書 |
|---|---|---|---|
| 1-1 | 改修項目を依頼（CR）単位に分割し、依頼ごとにトラック判定（[`README.md`](./README.md) §0）。C は要件定義の対象から外す | 起票者 | `00` メタ情報 |
| 1-2 | 全依頼分の `00` と、トラック A の `10`（表・例・権限）を AI に**一括起案**させる。AI は判断できない欄を空欄にせず「選択肢 + 仮案」で未決事項 Q 表に積む | AI | `00` / `10` draft |
| 1-3 | **レビュー会**（執筆会ではない。半日で依頼の束を処理する）: 顧客は §3 チェックリストの 5 判断 — 値の境界と端数 / 遷移の可否と権限 / エラー時の業務的扱い / 例の期待値 / 既存データの扱い — だけを確認・修正する | 顧客 + 起票者 | `10` §3〜§10 |
| 1-4 | 例・反例・境界例（`10` §10）の顧客承認 ✔ を埋める。**ここが後工程全部のテストオラクル** | 顧客 | `10` §10 |
| 1-5 | 決めきれない論点は Q 表に「仮決定 + 決定者 + 期限」で残す（会議を止めない） | 顧客 | `00` §7 |

**完了条件（フェーズゲート）**: 全依頼が approved / `10` §10 に顧客承認 ✔ / 未決事項がゼロまたは全件期限付き。

## 変更管理（フェーズ 2 以降の仕様変更）

- 変更は**新しい `00`（変更依頼）として起票**し、フェーズ 1〜2 と同じ承認を該当分だけ通す。元の依頼書は書き換えない（変更履歴に参照を追記）。
- 契約（openapi・イベント）に触れる変更は `20` §6 / `40` §7 の再承認が必須。oasdiff の破壊的変更ゲートが「申告なき契約変更」を機械的に止める。
- 影響範囲調査は人間がやらない: 変更差分を AI に渡し、FS0025（match 網羅）・生成コードドリフト・Pact の赤くなった箇所一覧を影響範囲として報告させる。
