# case-002 販売案件 — メタ情報

## 領域

販売案件ドメイン。直接販売・予約販売・委託販売の3形態それぞれが
独立したライフサイクルを持つ。

| メソッド | 内容 |

| `expert-review` | 主メソッド。生成 F# のレビュー基準で人手判定 |

## レビュー基準（expert-review チェックリスト）

### 構造
- [ ] `販売案件 = 直接販売案件 OR 予約販売案件 OR 委託販売案件` が判別共用体に翻訳されている
- [ ] 各販売案件のライフサイクル（直接販売は5状態、予約販売は4状態、委託販売は3状態）が状態ごとに型として表現されている
- [ ] `販売案件共通` レコードが各状態に共通フィールドとして含まれている
- [ ] `List<在庫ロット>` は `NonEmptyList<InventoryLot>` で表現されている

### 命名（dsl-conversion-rules.md の規約に従う）
- [ ] `直接販売案件` → `DirectSalesCase`（または同等の自然な英訳）
- [ ] `予約販売案件` → `ReservationSalesCase`
- [ ] `委託販売案件` → `ConsignmentSalesCase`
- [ ] 各状態 `〜済み` / `〜指示済み` 等の語幹が一貫している

### 振る舞い
- [ ] 全 12 個の `behavior` が関数として実装されている
- [ ] 入力型 / 出力型 / エラー型のシグネチャが DSL と一致
- [ ] エラーは `Result<Output, Error>` で返している
- [ ] エラー型のバリアントが業務的に意味のある粒度で構造化されている

### 設計原則
- [ ] Make Illegal States Unrepresentable（不正な状態遷移が型レベルで作れない）
- [ ] 純粋関数（副作用なし）
- [ ] 例外を投げていない

## このケースで検出したいこと（評価対象 B, C）

- リファレンスを見せた状態で、AI が **未知ドメインに翻訳規則を一般化できるか**（B. 規則の十分性）
- DSL で「価格査定」「販売契約」「予約価格」「出荷指示情報」など他領域の型を参照しているが、それらの内部構造は別ケース。
  AI がこれらを「依存型」として宣言できるか、抽象識別子として扱えるか（C. DSL の表現力）

## 既知の課題

- `behavior 販売案件を削除する -> 削除済み` の `削除済み` は単なる指示状態を表すマーカー型として実装する必要があり、DSL からは導けない（実装判断）
- `委託業者情報`, `委託販売結果`, `予約価格情報` などが DSL 内で未定義 → AI は branded type / placeholder として宣言する必要がある
